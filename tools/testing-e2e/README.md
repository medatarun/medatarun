# Automation e2e testing

## What it does

- Download a released version of Medatarun or build it from Gradle.
- Launches a series of end-to-end tests to ensure that the application starts
  correctly. For each test:
    - Create a Docker container with Medatarun (SQLite mode)
    - launch the same test using successively
        - API calls
        - CLI inside the container
        - CLI outside the container
    - tests will create bootstrap admin, create users, test login, etc.
- If using the "gradle" version
    - Launches all unit tests using SQLite
    - Launches all unit tests using PostgreSQL

## Prepare environment

Install `uv` first if it is not already available on your machine.
See https://docs.astral.sh/uv/

Then initialize a local environment for this project and install the test
dependencies:

```bash
cd tools/testing-e2e
uv venv
uv sync --extra test
```

Run the tests with:

```bash
uv run pytest
```

## Arguments

`testing-e2e.py` takes the following arguments:

- either `--from-gradle`
- or `--from-dist-url=<url>`

If none provided, the `--from-gradle` option is assumed

## Determining the artifact location

**If `--from-gradle` is used**

- if the `MEDATARUN_SRC` environment variable is defined, Gradle will be taken
  from here
- otherwise it is assumed relative to this project location

`$MEDATARUN_SRC` must point to the source code of the project where a
`./gradlew` file is present, otherwise it is an error.

Then we run `./gradlew build`. This produces
`$MEDATARUN_SRC/app/build/distributions/medatarun-dev.zip`

**If `--from-dist-url=<url>` is used**

then `<url>` is where an already
downloadable Medatarun `.zip` artifact can be found.

Archive will be downloaded and extracted. Tests will use that archive.

## Tests variants against the application

The application is always run in Docker with TestContainers.

The `.zip` artifact must be unzipped into `/opt/medatarun` (called
`$MEDATARUN_HOME` below).

Server will start with `$MEDATARUN_HOME/medatarun serve`

We know the application is running when the logs show:
`Starting REST API on http://0.0.0.0:8080 with publicBaseUrl=`

Tests are executing Medatarun actions. Each test against the container will be
run multiple times by changing the action execution mode:

- by executing CLI commands inside the container: `$MEDATARUN_HOME/medatarun`
- by executing CLI commands from outside the container (this tests remote CLI),
  in this case:
    - the `.zip` artifact is unzipped locally in a temporary directory (called
      `$MEDATARUN_TEST_CLI_HOME` here)
    - `$MEDATARUN_TEST_CLI_HOME/medatarun` is executed by passing the
      coordinates of the Docker container
- by using the API `/api/<actiongroup>/<action>`

Also, some tests use variants about bootstrapping too, because we need to check
the behavior when:

- no bootstrap secret had been provided to the container. In this scenario, the
  boostrap secret must be read
  from the Docker logs, by looking at the log message
  `BOOTSTRAP SECRET (one-time usage): $secret`, where `$secret` is the
  secret
- with a bootstrap secret provided to the container as the environment variable.
  `$MEDATARUN_AUTH_BOOTSTRAP_SECRET` is set with a random string generated by
  the test itself.

## Ensuring all unit tests pass on SQLite and PostgreSQL

Two "meta tests" have been added to ensure that all Gradle tests were run.
This is only done when the `--from-gradle` option is present.

To test on SQLite: `./gradlew test --rerun-tasks` is launched without
environment variable

To test with PostgreSQL `./gradlew test --rerun-tasks` is launched with
environment variable `MEDATARUN_STORAGE_DATASOURCE_JDBC_DBENGINE` set to
`postgresql` and that's enough to do the job.

This environment variable is understood by Medatarun as the regular
configuration option
`medatarun.storage.datasource.jdbc.dbengine=postgresql`.

Unit tests classes that use database have the `@EnableDatabaseTest` annotation.
It will trigger the `EnableDatabaseTestsExtension`.

This extension will either:

- read configuration and detect if the database is either SQLite or PostgreSQL
- create an empty in memory database for SQLite or launch a TestContainer Docker
  container with PostgreSQL inside, and grab
  the JDBC URL, user, password.
- will complete the `medatarun.storage.datasource.jdbc.url`, user and password,
  as System Properties based on the container information.
- drop and recreate the database.

When the test starts, it usually creates a full Plaform like a fresh
installation. When the Platform starts, all System Properties, now containing
the JDBC URL, user, password, are read as usual, so the application is tested as
it runs in an normal environment without any hacks.

This ensures that all tests run from a fresh database with the baseline
initialization scripts.

A bit heavy but safer.
